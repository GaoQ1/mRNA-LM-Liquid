代码链路要点

  • Makefile 中 train-hf 调用 python finetune_all.py --task halflife。
  • finetune_all.py 对 halflife 走 build_saluki_dataset，并用 Pearson/Spearman 做评估。
  • FullModel.py 使用 3 个预训练编码器（5'UTR、CDS、3'UTR），加 LoRA 微调与回归头。
  • 本任务每个样本是“完整序列 -> 一个标量标签”，不是“随时间推进的多步观测序列”。
关键判断： 液态网络的强项在连续时间/不规则采样/在线动态系统；而你现在的问题是离线静态序列回归，任务偏好与液态网络优势不完全重合。

2) 为什么“当前不建议”改成液态神经网络

A. 任务形态不匹配（核心）

  • 你当前不是流式时序预测，而是离线样本回归。
  • 液态网络的“输入驱动时间常数变化”在此类静态输入上，增益通常不如在真实时序任务明显。

B. 现有预训练资产价值很高

  • 当前方案依赖 3 个 RNA 领域预训练模型，这部分是主要性能来源之一。
  • 若全量换液态主干，通常意味着丢失现有预训练迁移优势，等于重新赌博。

C. 数据规模与风险收益比

  • 12k 级别数据并不算“大到能轻松支撑全新架构重学”。
  • 新架构带来更大的超参空间与稳定性风险，工程投入高，收益不确定。

D. 工程替换成本高于预期

  • 需要重写/重接模型定义、训练流程、推理加载和性能基准。
  • 与现有 API 的模型路径、参数管理、A/B 基准都会重新校准。

E. 当前先验瓶颈并不在“模型类型”

  • 基线里存在训练一致性问题（如 batch 参数未真正生效、best model 指标未完整接入等），先修这些通常比换架构更高效。
工程结论： 现阶段直接改液态网络属于“高改造成本 + 中高不确定收益 + 较高回归风险”，不满足优先级条件。

4) 决策建议（偏保守、可量化）

推荐策略：保留 Transformer 主干，先做基线修复 + 小范围验证

  • 先修现有训练一致性问题，再谈架构更替。
  • 若确实想试液态网络，建议仅替换回归头（不动 3 个预训练编码器）。
  • 设置清晰 Go/No-Go 门槛：
    • 验证集 Spearman 提升至少 0.02；
    • 测试集不退化（波动不超过 0.01）；
    • 训练时间不超过当前 1.3 倍；
    • 3 个随机种子方向一致。
最终结论： 对你当前的 train-hf/halflife 任务，Transformer 路线在“性能确定性 + 工程成本 + 交付风险”三者平衡上更优；液态网络应作为受控实验选项，而非主线替换方案。
